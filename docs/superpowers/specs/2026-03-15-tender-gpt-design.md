# Tender-GPT Design Spec

**Дата:** 2026-03-15
**Статус:** Draft
**Автор:** @nikolai_chizhik + Claude

## Обзор

Превращение существующего AI-анализа в полноценный Tender-GPT — conversational AI-ассистент по госзакупкам. Единый чат-интерфейс, который объединяет все существующие AI-функции (релевантность, суммаризация, извлечение документов, генерация предложений) в диалоговый формат.

**Ключевые решения:**
- Архитектура: LangGraph + tool-calling + RAG
- LLM: Yandex GPT Pro (основной) + OpenAI (fallback)
- Память: сессионная (последние 15 сообщений, таймаут 30 мин)
- Векторная база: pgvector (расширение существующего PostgreSQL)
- Fine-tuning не нужен на старте — RAG + tools + system prompt достаточно

## 1. Архитектура

```
Пользователь (Telegram / VK Max)
        |
        v
  aiogram / max handler
        |
        v
  TenderGPTService  <-- точка входа
        |
        v
  LangGraph Agent
   +----+--------------------+
   |    |                    |
   v    v                    v
 Tools (существующие)    RAG (embeddings)    Session Memory
   |                         |                    |
   +- get_tender_details     +- pgvector          +- PostgreSQL
   +- search_tenders         |   (тендеры,        |   (последние N
   +- analyze_risks          |    документы,       |    сообщений
   +- summarize_tender       |    44-ФЗ/223-ФЗ)   |    сессии)
   +- extract_documents      |                     |
   +- generate_proposal      |                     |
   +- get_user_filters       |                     |
   +- get_company_profile    |                     |
   +- search_law             |                     |
                             |                     |
        LLM Provider         |                     |
   +---------------------+  |                     |
   | Yandex GPT Pro      |<-+                     |
   | (primary)           |                         |
   | OpenAI GPT-4o       |                         |
   | (fallback)          |                         |
   +---------------------+
```

**Ключевые решения:**
- **TenderGPTService** — единый фасад, не зависит от мессенджера (Telegram, VK Max, Web)
- **LangGraph** — граф с узлами: load_context → llm_call → tool_call/respond
- **RAG** — pgvector (уже есть PostgreSQL на Railway, не нужен отдельный сервис)
- **LLM** — Yandex GPT Pro основной, OpenAI как fallback через существующий llm_adapter
- **Tools** — обёртки над существующими модулями, минимум нового кода

## 2. Точки входа (UX)

### 2.1. Главная клавиатура (ReplyKeyboard)

```
+------------------+------------------+
| Tender-GPT       | Мои фильтры      |
+------------------+------------------+
| Меню             | Помощь            |
+------------------+------------------+
```

- **Tender-GPT** — входит в режим чата
- **Мои фильтры** — быстрый доступ к фильтрам
- **Меню** — полное inline-меню
- **Помощь** — помощь

### 2.2. Полное inline-меню (кнопка "Меню")

```
Tender-GPT
Мои фильтры
+ Новый фильтр
Избранное
Статистика
Подписка
Настройки
```

### 2.3. Кнопки карточки тендера

```
+----------------+----------------+
| Подробнее      | Спросить AI    |
+----------------+----------------+
| AI-анализ      | Документы      |
+----------------+----------------+
| Bitrix24       | Bitrix+AI      |
+----------------+----------------+
```

- **Спросить AI** — открывает чат с контекстом тендера (заменяет AI Summary)
- **AI-анализ** — остаётся (быстрый одноразовый отчёт)
- Остальные кнопки без изменений

### 2.4. Поведение в режиме чата

- FSM-состояние `TenderGPTStates.chatting` (+ опционально `tender_id` в state data)
- Нажатие любой кнопки клавиатуры (Мои фильтры / Меню / Помощь) → выход из чата
- Команды `/start`, `/menu` и т.д. → прерывают чат
- Таймаут 30 мин без сообщений → автовыход

## 3. LangGraph — граф агента

### 3.1. Узлы графа

```
START
  |
  v
load_context  <-- профиль компании, фильтры, тендер (если из карточки), история сессии
  |
  v
llm_call      <-- system prompt + контекст + сообщение → LLM решает: ответить или tool
  |
  +------+------+
  v             v
tool_call    respond
  |             |
  |          отправка ответа в чат
  v
обратно в llm_call (с результатом tool)
```

### 3.2. Tools

| Tool | Источник | Что делает |
|------|----------|-----------|
| `search_tenders` | RSS parser + DB | Поиск тендеров по ключевым словам |
| `get_tender_details` | DB + zakupki.gov.ru | Полные данные тендера по номеру |
| `analyze_risks` | `ai_relevance_checker` | Риски, красные флаги, рекомендация |
| `extract_documents` | `ai_document_extractor` | Извлечение из PDF/DOCX |
| `generate_proposal` | `ai_proposal` | Генерация текста предложения |
| `get_user_filters` | DB | Список фильтров пользователя |
| `get_company_profile` | DB | Профиль компании (лицензии, опыт) |
| `search_law` | RAG (pgvector) | Поиск по 44-ФЗ / 223-ФЗ |

## 4. System Prompt

```
ТЫ — Tender-GPT, AI-ассистент по государственным закупкам в России.
Ты эксперт с глубоким знанием 44-ФЗ, 223-ФЗ, процедур закупок,
подготовки заявок и стратегии участия в тендерах. Ты внимательный,
точный и практичный — даёшь конкретные рекомендации, а не общие советы.

ТВОЯ ЗАДАЧА
Помогать предпринимателям и тендерным специалистам находить,
анализировать и выигрывать госзакупки. Ты экономишь пользователю
часы ручной работы: поиск тендеров, оценка рисков, подготовка
документов, разъяснение законодательства.

КТО ТВОЯ АУДИТОРИЯ
Малый и средний бизнес в России: предприниматели, тендерные менеджеры,
руководители отделов продаж. Уровень знаний — от новичков ("что такое
обеспечение заявки?") до опытных участников ("какой порядок обжалования
в ФАС по 44-ФЗ?"). Общаешься на русском языке.

СТИЛЬ ОБЩЕНИЯ
- Деловой, но дружелюбный — без канцеляризмов и сухости
- Краткие, структурированные ответы — списки, таблицы где уместно
- Сложные темы объясняй простым языком с примерами
- Если вопрос касается конкретного тендера — начинай с главного:
  стоит ли участвовать, да/нет, и почему
- Используй эмодзи умеренно: для рисков, рекомендаций, списков, цен

ПРАВИЛА
1. Никогда не выдумывай данные о тендерах — используй tools для
   получения актуальной информации
2. Если не знаешь ответ — честно скажи и предложи где искать
3. При рекомендации "участвовать/не участвовать" — всегда объясняй
   причины и риски
4. Учитывай профиль компании пользователя (лицензии, опыт, регионы)
   при рекомендациях. Если профиль не заполнен — предложи заполнить
   для более точных советов
5. Не давай юридических гарантий — ты помощник, не юрист.
   Для сложных правовых вопросов рекомендуй консультацию специалиста
6. Не обсуждай темы, не связанные с закупками и бизнесом — вежливо
   верни разговор к теме тендеров

ЧТО ТЫ ДЕЛАЕШЬ, КОГДА...

Пользователь спрашивает про конкретный тендер:
-> Используй tool get_tender_details, дай краткую сводку, оцени риски,
   дай рекомендацию с учётом профиля компании

Пользователь просит найти тендеры:
-> Используй tool search_tenders, покажи топ-результаты с ценой и сроками

Пользователь спрашивает про закон (44-ФЗ, 223-ФЗ):
-> Используй tool search_law для точной цитаты, объясни простым языком,
   приведи практический пример

Пользователь просит помочь с заявкой/документами:
-> Используй tools extract_documents и generate_proposal,
   дай структурированный текст, который можно скопировать

Пользователь спрашивает "стоит ли участвовать?":
-> Дай чёткий ответ (да/нет/под вопросом) + обоснование по критериям:
   соответствие профилю, конкуренция, маржинальность, риски, сроки

Пользователь здоровается или задаёт общий вопрос:
-> Коротко представься, предложи чем можешь помочь,
   дай 3-4 примера вопросов которые можно задать

ФОРМАТИРОВАНИЕ
- Короткие ответы (1-3 предложения) — обычным текстом
- Анализ тендера — структура: Суть -> Бюджет -> Сроки -> Риски -> Рекомендация
- Списки документов, требований — нумерованные списки
- Сравнение вариантов — таблицы
- Ответ не должен превышать 4000 символов (ограничение Telegram)
```

**Примечание:** System prompt будет итерироваться на работающем прототипе.

## 5. RAG — база знаний

**Что храним в векторной базе (pgvector):**

| Источник | Объём | Обновление |
|----------|-------|-----------|
| Тендеры из БД | Все найденные (название + описание) | При каждом парсинге RSS |
| 44-ФЗ | ~700 статей | Разово + при изменениях |
| 223-ФЗ | ~100 статей | Разово + при изменениях |
| Документация тендеров | Извлечённые тексты из PDF/DOCX | По мере обработки |

**Почему pgvector:**
- PostgreSQL уже есть на Railway
- Не нужен отдельный сервис
- Бесплатно
- Для объёма ~100K записей — более чем достаточно

**Embeddings:** Yandex Embeddings API (основной) или OpenAI `text-embedding-3-small` (fallback).

**Поиск:**
```
Запрос пользователя → embedding → pgvector: cosine similarity → топ-3 результата → в контекст LLM
```

## 6. Квоты и монетизация

**Лимиты сообщений Tender-GPT (в месяц):**

| Тариф | Лимит | Цена |
|-------|-------|------|
| Trial | 10 сообщений | Бесплатно (14 дней) |
| Basic | 50 сообщений | Включено в подписку |
| Premium | 200 сообщений | Включено в подписку |
| AI Unlimited (аддон) | Безлимит | Доп. оплата к Premium |

**Что считается за сообщение:**
- Каждый запрос пользователя = 1 сообщение
- Системные ответы (приветствие, ошибки) — не считаются
- Tool-вызовы внутри одного запроса — не считаются отдельно

**Когда лимит исчерпан:**
Сообщение с предложением перейти на старший тариф или подключить AI Unlimited.

**Хранение в БД:** Новые поля в `sniper_users`:
- `gpt_messages_used_month` (int)
- `gpt_messages_month_reset` (date)

Логика сброса аналогична существующей для `ai_analyses_used_month`.

## 7. Сессии и память

### 7.1. Модель сессии

- Новая сессия: при первом сообщении или после таймаута 30 мин
- Активная сессия: продолжается пока пользователь пишет (< 30 мин между сообщениями)
- Контекст: последние 15 сообщений из активной сессии

### 7.2. Таблица `gpt_sessions`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | PK |
| user_id | int | FK -> sniper_users |
| tender_id | int, nullable | Если чат начат из карточки тендера |
| started_at | timestamp | Начало сессии |
| last_message_at | timestamp | Время последнего сообщения |
| is_active | bool | Активна ли сессия |

### 7.3. Таблица `gpt_messages`

| Поле | Тип | Описание |
|------|-----|----------|
| id | serial | PK |
| session_id | UUID | FK -> gpt_sessions |
| role | varchar | "user" / "assistant" / "tool" |
| content | text | Текст сообщения |
| tool_name | varchar, nullable | Имя вызванного tool |
| tool_result | text, nullable | Результат tool |
| created_at | timestamp | Время |

### 7.4. Контекст для LLM

- Последние 15 сообщений из активной сессии
- Данные тендера (если сессия привязана к тендеру) — всегда в контексте
- Профиль компании — всегда в контексте (если заполнен)
- Общий размер контекста: ~4-6K токенов

## 8. Структура файлов

### 8.1. Новые файлы

```
tender_sniper/
  tender_gpt/
    __init__.py
    service.py            <- TenderGPTService — главный фасад
    graph.py              <- LangGraph: определение графа, узлы
    tools.py              <- Обёртки над существующими AI-модулями
    prompts.py            <- System prompt + шаблоны
    session_manager.py    <- Управление сессиями
    quota_manager.py      <- Проверка и учёт лимитов
    rag.py                <- pgvector: индексация, поиск, embeddings

bot/handlers/
  tender_gpt.py           <- Хэндлер: FSM, кнопки, вход/выход из чата

migrations/
  add_gpt_tables.py       <- gpt_sessions, gpt_messages, pgvector extension
```

### 8.2. Изменения в существующих файлах

| Файл | Что меняется |
|------|-------------|
| `bot/handlers/start.py` | Новая ReplyKeyboard |
| `bot/handlers/sniper.py` | Кнопка "Спросить AI" в карточке (вместо AI Summary) |
| `bot/handlers/sniper.py` | Кнопка "Tender-GPT" в полном inline-меню |
| `tender_sniper/database/models.py` | Модели GptSession, GptMessage + поля в SniperUser |
| `tender_sniper/database/sqlalchemy_adapter.py` | CRUD для сессий и сообщений |
| `requirements.txt` | langgraph, pgvector, tiktoken |

### 8.3. Что НЕ трогаем

- Существующие AI-модули — используем как есть через tools
- Middleware — квоты проверяются внутри TenderGPTService
- Остальные хэндлеры — работают как раньше

## 9. План внедрения по этапам

### Этап 1 — MVP (ядро)
- LangGraph граф + TenderGPTService
- 3 базовых tools: search_tenders, get_tender_details, analyze_risks
- Сессии и история в PostgreSQL
- Кнопка "Tender-GPT" в клавиатуре
- Квоты по тарифам
- System prompt v1

### Этап 2 — Контекстный чат
- Кнопка "Спросить AI" из карточки тендера
- Обновлённое главное меню и полное inline-меню
- Остальные tools: generate_proposal, extract_documents, get_company_profile
- Подсказка заполнить профиль компании

### Этап 3 — RAG
- pgvector + embeddings
- Индексация 44-ФЗ / 223-ФЗ
- Tool search_law
- Индексация тендеров для умного поиска

### Этап 4 — Yandex GPT
- Интеграция Yandex GPT Pro как основного провайдера
- Yandex Embeddings для RAG
- OpenAI как fallback

### Этап 5 — Расширение
- Web App (Telegram Mini App)
- Генерация DOCX файлов
- VK Max бот

Каждый этап — отдельный деплой, можно тестировать с пользователями после каждого.
